Oracle PL/SQL: Package-Level Data

Question

I execute this statement:

CREATE OR REPLACE PACKAGE plch_pkg
IS
   my_resolution VARCHAR2(30) := 'Turn off the TV';
END;
/

Which of the choices display "Turn off the TV" after execution?

Choice 1 is correct. You and 73% others got it right!

BEGIN
   EXECUTE IMMEDIATE
      'begin dbms_output.put_line (plch_pkg.my_resolution); end;';
END;
/

Sure, this works just fine. I execute a dynamic PL/SQL block and inside that block reference the package variable.

Choice 2 is correct. You and 88% others got it right!

BEGIN
   DBMS_OUTPUT.put_line (plch_pkg.my_resolution);
END;
/

Eh, yup. That definitely does the trick, too!

Choice 3 is not correct. You got it wrong, while 63% got it right.

SELECT plch_pkg.my_resolution FROM DUAL
/

Fails with

ORA-06553: PLS-221: 'MY_RESOLUTION' is not a procedure or is undefined

because you cannot directly reference a package variable inside a SQL statement - at least, one that executes outside of a PL/SQL block.

Choice 4 is not correct. You and 61% others got it right!

DECLARE
   l_resolution   plch_pkg.my_resolution%TYPE;
BEGIN
   EXECUTE IMMEDIATE 'select plch_pkg.my_resolution from dual'
      INTO l_resolution;

   DBMS_OUTPUT.put_line (l_resolution);
END;
/

Fails with

ORA-06553: PLS-221: 'MY_RESOLUTION' is not a procedure or is undefined

because you cannot directly reference a package variable inside a SQL statement - at least, one that executes outside of a PL/SQL block.

"But, wait!" you say, "that IS executed inside a PL/SQL block!"

Yes and no. The dynamic SQL statement is passed to the SQL engine from the PL/SQL block, but the statement itself is a stand-alone SQL statement.

It executes outside of the PL/SQL block context.

Choice 5 is correct. You and 67% others got it right!

DECLARE
   l_resolution   plch_pkg.my_resolution%TYPE;
BEGIN
   SELECT plch_pkg.my_resolution INTO l_resolution FROM DUAL;

   DBMS_OUTPUT.put_line (l_resolution);
END;
/

Now the SQL statement executes from within a block and so there is no problem referencing the package variable.

Explanation

Summary

Variables and constants declared in the package specification cannot be directly referenced inside a SQL statement that is executed outside of a PL/SQL block.


Background

Oracle PL/SQL allows you to construct applications based on program units of various types including the basic block (both anonymous and nested), functions, procedures, packages, triggers and object types.

A package is a PL/SQL program unit, defined in the database, that groups together logically related PL/SQL types, variables, constants, subprograms (procedures and/or functions), cursors, and exceptions.

A package always has a specification, which declares the items that can be referenced from outside of the package. If the public items include cursors that do not include their SELECT statement (only a RETURN clause) or subprograms, then the package must also have a body.

Packages are the fundamental building blocks of PL/SQL-based applications.

When you declare a variable or constant in a package outside of any of the package's subprograms, that is called package-level data. Here's a package that has a variable declared in the package body and a constant in the package specification:

CREATE OR REPLACE PACKAGE plch_pkg
IS
   c_max_varchar2   CONSTANT PLS_INTEGER := 32767;

   PROCEDURE set_private_variable (number_in IN NUMBER);
END;
/

CREATE OR REPLACE PACKAGE BODY plch_pkg
IS
   g_number   NUMBER;

   PROCEDURE set_private_variable (number_in IN NUMBER)
   IS
   BEGIN
      g_number := number_in;
   END;
END;
/

Notice that g_number is inside the package body. As a result, a session can only change the value of that variable by calling the set_private_variable procedure.

Some key things to remember about package-level data:

  • Memory for local variables is automatically released when the block in which they are declared terminates, but memory for package-level data is not released until you disconnect, recompile the package or take some other action to cause the package to be reinitialized.
  • As a consequence, package-level data is persistent between server calls to the package. If in one block you assign a value to a package-level variable, when another block executes in the same session, that variable will still have that value.
  • Memory for package-level data is stored in the UGA (user global area), which is a per-session memory area. You should therefore be very careful about declaring potentially "large" structures like collections at the package level. You could consume lots of memory without realizing it.
