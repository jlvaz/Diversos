Oracle PL/SQL: Deterministic Functions

Question

I execute the following statements:

CREATE TABLE plch_trees
(
   id   NUMBER,
   nm   VARCHAR2 (100)
)
/

CREATE OR REPLACE PACKAGE plch_pkg
IS
   my_global   NUMBER;
END;
/

Which of the choices compile successfully?

Choice 1 is correct. You and 68% others got it right!

CREATE OR REPLACE FUNCTION plch_func (id_in IN INTEGER)
   RETURN VARCHAR2
   DETERMINISTIC
IS
BEGIN
   UPDATE plch_trees
      SET nm = UPPER (nm)
    WHERE id = id_in;

   RETURN TO_CHAR (id_in);
END;
/

This function is not "pure" of side effects, which a deterministic function should be, since it contains an UPDATE statement. That will not, however, stop the compiler from setting the status of this function to VALID.

Choice 2 is correct. You and 66% others got it right!

CREATE OR REPLACE FUNCTION plch_func (id_in IN INTEGER)
   RETURN VARCHAR2
   DETERMINISTIC
IS
BEGIN
   RETURN TO_CHAR (plch_pkg.my_global);
END;
/

This function is not "pure" of side effects, which a deterministic function should be, since it contains an out of scope reference to the package variable. That will not, however, stop the compiler from setting the status of this function to VALID.

Choice 3 is correct. You got it wrong, while 79% got it right.

CREATE OR REPLACE FUNCTION plch_func (id_in IN INTEGER)
   RETURN VARCHAR2
   DETERMINISTIC
IS
   l_nm  plch_trees.nm%TYPE;
BEGIN
   SELECT nm
     INTO l_nm
     FROM plch_trees
    WHERE id = id_in;

   RETURN TO_CHAR (id_in);
END;
/

This function is not "pure" of side effects, which a deterministic function should be, since it contains a query. That will not, however, stop the compiler from setting the status of this function to VALID.

Choice 4 is correct. You and 90% others got it right!

CREATE OR REPLACE FUNCTION plch_func (id_in IN INTEGER)
   RETURN VARCHAR2
   DETERMINISTIC
IS
BEGIN
   RETURN TO_CHAR (id_in);
END;
/

This version of the function actually is pure of side effects; it is a deterministic function. And the PL/SQL compiler happily accepts the designation (which, as the other choices show, it will do regardless).

Explanation

Summary

The DETERMINISTIC keyword is useful for self-documentation of code (you are expressing an intention, that this function should have no side-effects). But the PL/SQL compiler does not check to see if the function truly is deterministic.


Background

Oracle PL/SQL allows you to construct applications based on program units of various types including the basic block (both anonymous and nested), functions, procedures, packages, triggers and object types.

A deterministic function is a function whose outcome (the value returned) is determined solely by the values passed into it as arguments. Another way to put it is that the function has no side-effects, such as referencing or modifying the contents of package variables or the database.

Here is an example of a deterministic function:

FUNCTION double_the_fun (n NUMBER)
   RETURN NUMBER
IS
BEGIN
   RETURN n * 2;
END;

and here is a non-deterministic function:

FUNCTION double_the_fun (n NUMBER)
   RETURN NUMBER
IS
BEGIN
   DELETE FROM employees;
   RETURN n * 2;
END;

You can declare a function to be deterministic by adding that keyword to the header as in:

FUNCTION double_the_fun (n NUMBER)
   RETURN NUMBER DETERMINISTIC
IS
BEGIN
   RETURN n * 2;
END;

So why bother declaring your function as determinstic? Because under certain circumstances, Oracle will optimize execution of the function. It does so by remembering the values of arguments and return values for executions of the function. If your session then calls the function with the same argument values, Oracle knows (since the function is deterministic) what to return without executing the function.

These circumstances include:

  • Calling the function inside a SQL statement.
  • Calling the function in a PL/SQL block (Oracle Database 11g and higher) - and passing compile-time static values for all IN arguments.

In addition, if you want to define a function-based index, that function must be declared as deterministic.

Note that Oracle will not attempt to validate your claim of determinism. In other words, you can lie to Oracle and declare a function as deterministic even when it is not. This can lead to all sorts of problems and should be avoided.
