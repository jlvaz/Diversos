Question

I have a table of employees:

create table plch_employee (
   empno    number(4)   primary key
 , ename    varchar2(10)
 , sal      number(7,2)
 , comm     number(7,2)
)
/

insert into plch_employee values (7369, 'SMITH' ,  800, null)
/
insert into plch_employee values (7499, 'ALLEN' , 1600,  300)
/
insert into plch_employee values (7521, 'WARD'  , 1250,  500)
/
insert into plch_employee values (7566, 'JONES' , 2975, null)
/
insert into plch_employee values (7654, 'MARTIN', 1250, 1400)
/
insert into plch_employee values (7698, 'BLAKE' , 2850, null)
/
insert into plch_employee values (7782, 'CLARK' , 2450, null)
/
insert into plch_employee values (7788, 'SCOTT' , 3000, null)
/
insert into plch_employee values (7839, 'KING'  , 5000, null)
/
insert into plch_employee values (7844, 'TURNER', 1500,    0)
/
insert into plch_employee values (7876, 'ADAMS' , 1100, null)
/
insert into plch_employee values (7900, 'JAMES' ,  950, null)
/
insert into plch_employee values (7902, 'FORD'  , 3000, null)
/
insert into plch_employee values (7934, 'MILLER', 1300, null)
/
commit
/

Boss would like a list of employees who have a salary of at least 1500 and at the same time a commission of at most 1000. (Boss here considers a null commission to qualify for that criteria.)

I have this incomplete query prepared for this:

select empno
     , ename
     , sal
     , comm
  from plch_employee
 /* WHERE CLAUSE */
 order by empno
/

Which of the choices can replace the comment /* WHERE CLAUSE */ to make the query return this desired output:

     EMPNO ENAME             SAL       COMM
---------- ---------- ---------- ----------
      7499 ALLEN            1600        300
      7566 JONES            2975
      7698 BLAKE            2850
      7782 CLARK            2450
      7788 SCOTT            3000
      7839 KING             5000
      7844 TURNER           1500          0
      7902 FORD             3000

Choice 1 is not correct. You and 95% others got it right!

 where sal >= 1500
   and comm <= 1000
Will not include rows with null comm.

Choice 2 is correct and recommended. You and 94% others got it right!

 where sal >= 1500
   and (comm <= 1000 or comm is null)

Includes rows with null comm.

Choice 3 is not correct. You got it wrong, while 85% got it right.

 where sal >= 1500
   and comm <= 1000
    or comm is null
Includes ALL rows with null comm, not just those with sal >= 1500, so this retrieves too many rows.

Choice 4 is correct but not recommended. You got it wrong, while 92% got it right.

 where sal >= 1500
   and nvl(comm,0) <= 1000
Treats null comm as if it were zero comm.

Choice 5 is not correct. You and 77% others got it right!

 where sal >= 1500
   and not (comm > 1000)
Condition is unknown, NOT condition is still unknown.

Choice 6 is correct. You got it wrong, while 55% got it right.

 where sal >= 1500
   and lnnvl(comm > 1000)

LNNVL returns TRUE if condition is FALSE or UNKNOWN.

Explanation

Summary

NULL values make conditions NULL (unknown) as well, which is neither TRUE nor FALSE. So in your WHERE clauses you typically need IS NULL or IS NOT NULL to specify whether to include NULL values as satisfying your criteria or not.


Background

A condition that evaluates to UNKNOWN acts almost like FALSE. For example, a SELECT statement with a condition in the WHERE clause that evaluates to UNKNOWN returns no rows. However, a condition evaluating to UNKNOWN differs from FALSE in that further operations on an UNKNOWN condition evaluation will evaluate to UNKNOWN. Thus, NOT FALSE evaluates to TRUE, but NOT UNKNOWN evaluates to UNKNOWN.

When comm is null, evaluating condition comm <= 1000 is neither true, nor false, it is unknown ("boolean null".) Evaluating condition not (comm > 1000) is also unknown. The WHERE clause needs to evaluate to true, it will not include rows that evaluate to unknown.

You can fix this like choice 4, which treats null values as if they were zero. Or you can use the IS NULL construct like choice 2. Or there is the "logical negation" LNNVL function of choice 6 that will return true if the inner condition evaluates to false or unknown.

Using choice 2 gives the optimizer valuable information that you are interested in both rows with comm less than or equal to 1000 as well as rows with null comm, so the optimizer has a much better chance of calculating correct cardinality using this method. If you have an index that includes comm and a not null column (or a constant) the optimizer can even use that index for choice 2.

You could improve choice 4 with extended statistics or a function based index on nvl(comm,0), then the optimizer also has information to help it reach the best decision. The downside is that choice 4 really treats null comm as if it were zero, so perhaps you might be better off by updating null comm values to zero and making the column not null instead - that would also give the optimizer much better information to work with rather than the NVL "trick".
